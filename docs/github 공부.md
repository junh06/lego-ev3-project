# 1. Git과 Github의 사용
이번 일회용컵 자동 분리수거기를 진행하며 소스 코드와 개발 과정 등을 팀원들과 공유하기 위해 Git과 Github을 사용하였다. Git을 활용하여 내 컴퓨터(local)에서 코드의 수정 내역을 단계별로 기록(Commit)하여 관리했고, Github를 통해 팀원들과 코드를 공유하였다.

# 2. Git의 작업 흐름과 3단계 구조
Git은 파일을 단순히 저장하는 것이 아니라 3가지 영역을 거쳐 저장된다는 점을 배웠다. 이 구조를 이해하고 각 단계에 맞는 명령어를 사용하는 것이 중요하다
# 2-1. 작업 디렉토리
내(사용자)가 vs code에서 코드를 실제로 작성하고 수정하는 공간이다. 이곳에서 파일이 생성되거나 수정되면 Git은 이를 추적되지 않음(UnTracked) 혹은 수정됨(Modified) 상태로 감지한다. - 파일 옆에 U나 M표시가 뜬다.
# 2-2. 스테이징 영역
커밋(저장)하기 전, 저장할 파일들만 골라서 임시로 올려두는 공간이다. 나는 작업한 모든 파일을 스테이징하기 위해 아래의 명령어를 사용하였다. 
git add . - 터미널에 작성해야 한다.
왜 바로 저장하지 않고 스테이징을 거칠까?라는 의문이 들었다. 그 이유로는 수정한 파일이 10개라도 이번에는 통신 기능 관련 파일 3개만 묶어서 기록하고 싶을 수 있다. 즉, 스테이징 영역은 논리적으로 연관된 변경 사항만 묶어서 커밋하기 위환 완충지대 역할을 한다.
# 2-3. 저장소
스테이징 된 파일들을 영구적인 기록으로 저장하는 곳이다. 이때 -m을 사용하여 어떤 작업을 했는지 메시지를 남길 수 있다.
git commit -m "프로젝트 코드 업로드"

# 3. 브랜치
메인 코드를 직접 건드리지 않고 안전하게 작업하기 위해 브랜치 기능을 활용하였다.
# 3-1. 개념과 활용
브랜치는 프로젝트의 원본을 복사해서 나만의 평행 우주를 만드는 개념이다. 아래 명령어를 통해 내 학번과 이름이 포함된 독립적인 작업 공간을 만들고 이동했다.
브랜치 생성
git branch 학번_이름
해당 브랜치로 이동
git checkout 학번_이름
이 공간에서 코드를 마음껏 수정하거나 실수하더라도 원본(main)에는 전혀 영향을 주지 않기 때문에 안심하고 개발할 수 있었다.
# 3-2. Pull Request
브랜치에서 작업이 끝나면 아래 명령어로 원격 저장소(Github)에 업로드 한다.
git push origin 학번_이름
하지만 바로 합쳐지는 것이 아니었다. Github 웹사이트에서 Compare & pull request 버튼을 눌러야 한다.

# 4. 트러블 슈팅
Git을 처음 사용하면서 겪은 문제들과 해결한 과정을 정리하였다.
# 4-1. 사용자 정보 누락 오류
문제 상황: git commit 명령을 입력했을 때 Please tell me who you are. 라는 에러가 발생하며 진행되지 않음

원인 분석: Git은 버전 관리 도구이기 때문에 누가 코드를 수정했는지에 대한 기록이 필수적이다. 초기 설정 시 작성자 정보가 등록되지 않아서 발생했다.

해결 명령어: git config --global user.name "내이름", git config --global user.email "내이메일"
# 4-2. 원격 저장소 중복 연결 오류
문제 상황: git remote add origin 주소 입력시 error: remote origin already exists 에러 발생.

원인 분석: origin 이라는 이름은 원격 저장소 주소의 별칭이다. 이미 이 이름으로 연결된 주소가 있거나 이전에 잘못된 주소로 연결을 시도한 기록이 남아있어서 발생했다.

해결 명령어: 기존의 잘못된 연결을 끊고, 올바른 주소로 다시 연결했다.

기존 연결 삭제
git remote remove origin

올바른 주소로 재연결
git remote add origin github레포지토리 주소(초록색[code] 버튼 눌러서 주소 복사)

# 5. 결론 및 느낀 점
이번 학습을 통해 Git은 단순한 저장소가 아님을 깨달았다. 터미널을 통해 명렁어를 직접 입력하며 파일의 상태 변화를 눈으로 확인했고, 브랜치를 통해 독립적으로 개발하고 PR을 통해 안전하게 합치는 과정을 익혔다. 이러한 버전 관리 역량은 앞으로 진행할 더 큰 규모의 프로젝트에서 필수적일 것이라고 생각한다.
